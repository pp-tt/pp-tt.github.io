Java 多线程

Posted on 2022-03-17 Edited on 2022-03-19 In Java

对 Java 多线程问题简单梳理总结。

1 相关概念

1.1 并行和并发

  • 并行：同一时刻，有多个指令在多个CPU上同时执行
  • 并发：同一时刻，有多个指令在单个CPU上交替之行

1.2 进程和线程

  • 进程： 是正在运行的软件
    • 独立性
    • 动态性
    • 并发性
  • 线程：是进程中单个顺序控制流，是一条执行路径

1.3 多线程的实现方案

  • 继承 Thread 类的方式进行实现
  • 实现 Runnable 接口的方式进行实现
  • 利用 Callable 和 Future 接口方式实现

2.2 相关问题

  • 为什么要重写 run() 方法？

    因为多线程在开启之后，实质上执行的就是 run() 方法内的内容，所以要重写 run() 方法，将其内包括的内容换成我们需要执行的内容。

  • run() 方法和 start() 方法的区别？

    run() 方法相当于普通的调用方法，即创建实例，然后调用 run() 方法，相当于直接调用；start() 是启动线程，然后由 JVM 来调用该线程的 run() 方法。

5 三种多线程方式区别

方式 优点 缺点
Thread 实现简单，可以直接调用 Thread 类中的其他方法 扩展性差，不能再继承其他的类
Runnable 扩展性好 实现较复杂，需要实现重写 Runnable 中的 run() 方法然后作为构造方法参数传递 Thread

7 线程安全问题

7.1 同步代码块

  • 同步技术原理

    使用了一个锁对象， 这个锁对象叫同步锁，也叫对象锁，也叫对象监视器。假设三个线程 t0、t1、t2 一起抢夺CPU的执行权，说抢到了谁执行 run 方法， t0 抢到了，执行 run 方法，遇到 synchronized 代码块，这是 t0 会检查 synchronized 代码块是否有锁对象，发现有，就会获取到锁对象进入到同步中执行。t1 抢到 CPU 执行权时也会判断是否有锁对象，这时锁对象在 t0 那，发现没有锁对象，t1会进入阻塞状态，一直等待 t0 将锁对象归还给同步代码块，t1 才能获取锁对象进入到同步中执行。

  • 同步技术缺点

    程序频繁的判断锁、获取锁、释放锁，程序的效率会降低。

9.2.3 volatile 关键字

  • 问题

    如果 A 线程修改了对中共享变量的值，那么其他线程不一定能及时使用最新的值。

  • 原理

    在 JVM 中堆内存是唯一的，每一个线程都会有自己的线程栈，每一个线程在使用堆里面变量的时候，都会先拷贝一份到变量的副本中，在线程中，每一次使用是从变量的副本中获取的。

  • 解决方案

    volatile 关键字：强制线程每次使用的时候，都会看一下共享区域最新的值，但是不能保证原子性。

    synchronized ：将涉及共享变量的代码放入同步代码块中。

  • 使用方法

    1. 在共享变量前加 volatile 关键字就好了
    2. 将涉及共享变量的代码放入同步代码块中。

  • synchronized 为什么能解决？

    同步代码块执行步骤如下：线程先获得锁，然后清空变量副本，拷贝共享变量最新的值到变量副本中，然后执行代码，将修改后变量副本中的值赋给共享数据，然后释放锁。

10 线程原子性

10.1 概念

所谓的原子性是指在一次操作或多次操作中，要么所有的操作全部得到了执行，并且不会收到任何因素的干扰而中断，要么所有的操作都不执行，多个操作是一个不可分割的整体。

10.2 解决方案

  • synchronized

    synchronized 能保证每次执行时都只有一个线程，其余线程进不了。缺点：频繁获取锁释放锁影响程序执行效率

  • AtomicInteger(针对整型，也有其他类型的可用)

10.3 AtomicInteget 原理

  • 原理

    CAS算法 + 自旋

  • CAS 算法：（内存值，旧的预期值，要修改的值）

    1. 若旧的预期值 == 内存值 则修改成功，将内存值改为要修改的值
    2. 若旧的预期值 != 内存值 则修改失败，则内存值先不修改，然后重新获取最新的内存值作为旧的预期值（自旋）

  • 通俗理解：

    CAS 算法就是在修改共享数据的时候，把原来的旧值记录下来了。如果现在内存中的值跟原来旧值一样，证明没有其他线程操作过内存值，则修改成功。

    如果现在内存中的值和原来的旧值不一样了，证明已经有其他的线程操作过了内存值，则修改失败，需要获取现在最新的值，再次进行操作，这个重新获取就是自旋

10.4 synchronized 和 CAS 区别（悲观锁和乐观锁）

  • 相同点

    在多线程情况下，都可以保证共享数据的安全性。

  • 不同点

    synchronized 总是从最坏的角度出发，认为每次获取数据的时候别人都有可能修改。所以在每次操作共享数据之前都会上锁（悲观锁）

    cas 是从乐观的角度出发，假设每次获取数据别人都不会修改，所以不会上锁。只不过在修改共享数据的时候，回去检查一下别人有没有修改过这个数据。如果别人修改过，那么就再次获取现在的最新值（自旋）。如果别人没有修改过，那么就直接修改共享数据的值。

11 并发工具类

11.1 Hashtable

HashMap 是线程不安全的（多线程环境下可能会存在问题）。为了保证数据安全性，可以使用Hashtable，但是效率低下。效率低下的具体原因如下：

  1. Hashtable 采取悲观锁 synchronized 的形式保证数据的安全性。
  2. 只要有线程访问，会将整张表全部锁起来，不让其余线程访问，所以 Hashtable 的效率低下。

  • ConcurrentHashMap 1.7 版本原理解析：

  • ConcurrentHashMap 1.8 版本原理解析：

    1. 如果使用空参构造创建 ConcurrentHashMap 对象，则什么事情都不做。在第一次添加元素的时候创建哈希表
    2. 计算当前元素应该存入的索引
    3. 如果索引位置为 null，则利用 CAS 算法，将本节点添加到数组中
    4. 如果索引位置不为 null，则利用 volatile 关键字获得当前位置最新的节点地址，挂在他下面变成链表
    5. 当链表长度大于等于8时，自动转换为红黑树
    6. 当链表或者红黑树头节点为锁对象，配合局部悲观锁保证多线程操作集合时数据的安全性
